# 2026-04-09 · v1.2.0

今天把 Phase 15 收成了一个我自己会明显感觉“系统开始更会怀疑自己”的版本。前面做到 `1.1.0` 的时候，我已经觉得这个项目像一个真实交付平台了，但它更像是一个会把结果整理清楚、解释清楚、交给你的系统。到了今天，我更在意的是，它能不能在交给你之前先停下来问一句：这个东西真的够格吗？

这一轮里最关键的，不是多了多少工具，也不是 UI 又多了几个字段，而是 acceptance proof 这个边界终于立起来了。一个 delivery 被产出，不代表它就应该被交付；它还要经过 verifier 这一层，才能变成“可接受的交付”。这种分层会让系统看起来更慢一点、更挑剔一点，但我反而觉得这更像真正能长期使用的东西。因为很多时候，问题不在于 agent 会不会做，而在于它会不会在该拒绝的时候拒绝。

今天我最有感觉的地方，是 research 和 browser 两条线终于开始在“验收”这件事上有了各自的性格。research 的 verifier 更像一个不太容易被骗的编辑，盯着 source coverage、references artifact、verification adequacy；browser 的 verifier 更像一个执拗的操作审查员，只认 outcome 有没有真的到位、recovery 是否已经耗尽。它们共用同一个 harness，但不会再说同一种空泛语言，这点让我很满意。

另一个让我觉得舒服的点，是 inspector 终于不仅能展示“发生了什么”，还能展示“为什么 verifier 还不肯放行”。我一直不太喜欢那种只给出一个 blocked 或 failed 的系统状态，它太像把责任又丢回给用户。现在至少可以明确告诉人：是接受了、是要求修订，还是直接拒收；还有 findings 和 summary 可以追。这个变化看起来不大，但我觉得它会直接影响一个人是否愿意相信这个系统。

发 `1.2.0` 的感觉和前两个版本又不太一样。`1.0.0` 像是产品面出来了，`1.1.0` 像是系统开始按任务族思考真实完成条件，而 `1.2.0` 更像是系统终于愿意站在交付门口做守门人。这种气质上的变化，比单个功能更重要。

今天整体状态还是比较稳，更多是那种需要反复校验边界和措辞的工作。让我觉得值的是，最后收出来的不是“又多了一层检查”，而是“交付和验收终于开始被明确区分”。这件事会让后面的很多阶段更有抓手。
