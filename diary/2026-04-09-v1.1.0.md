# 2026-04-09 · v1.1.0

今天把 Phase 14 收成了一个我自己会更愿意称之为“真实交付平台”的版本。和前几个阶段相比，这一轮最明显的变化不是内核又抽象得更漂亮了，而是系统开始真正按任务族去理解“什么叫完成”。研究写作和浏览器流程以前都能跑，但它们对“完成”的要求完全不同。今天最重要的事情，就是把这种差异从脑子里的判断，变成 runtime、inspection 和 UI 都能共同表达的事实。

写这轮的时候，我一直在反复提醒自己，不要把所有任务都塞进同一个交付模板里。研究写作最怕的是看起来像成品，但证据其实不够；浏览器流程最怕的是操作执行了一堆，但最后根本没达到页面结果。所以我让 research 去强调 source coverage、references、可信阻断；让 browser 去强调 step、validation、recovery。做到这里的时候，整个系统给人的感觉终于开始从“聪明的 runtime”往“真的能交东西的系统”挪了一步。

我今天最在意的一个小点，其实是 `runProofSummary`。它不复杂，也不炫技，但它把“系统做过什么”提炼成了一个人可以快速看懂的摘要。我越来越觉得，真实世界里的 agent 产品如果不能清楚回答“你到底做了什么”，那它再强也很难让人信任。技术上我们当然还能继续往深处做，但这种把事实压缩成一眼能懂的能力，本身就是产品力。

还有一个挺明显的感受是，这一轮比前面更像在做“区别化体验”，而不是单纯推进统一架构。统一很重要，但有些地方如果硬统一，最后只会把任务本身的真实性磨平。今天让我觉得比较踏实的是，research 和 browser 两条线虽然共用 delivery harness，但在用户看到的层面，已经开始各自长出自己的语言了。

发 `1.1.0` 的时候，我对这个项目的感觉和发 `1.0.0` 已经不太一样了。`1.0.0` 更像是“产品面成形”，而 `1.1.0` 更像是“系统开始理解不同任务的真实完成条件”。这件事不会一下子把所有真实任务都打通，但它像是一个比较清晰的转折点。

今天的工作状态是稳的，没有特别大的情绪起伏，更多是在做那种需要反复校准语言和行为一致性的工作。代码、解释、UI、交付证明，这几样东西只要有一个说了谎，整个系统的可信度就会立刻掉下去。把它们尽量拉齐，会让我觉得这一天做得值。
