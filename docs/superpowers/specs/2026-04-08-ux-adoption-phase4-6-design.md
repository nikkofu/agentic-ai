# UX & Adoption First 路线设计（Phase M4-M6）

- 日期：2026-04-08
- 当前基线：Phase 1 / 2 / 3 已完成（以 README 与规划文档为准）
- 目标周期：近 3 个月（按月里程碑）
- 目标优先级：采用率优先（UX & Adoption First）
- 交付节奏：每月一个里程碑（M4/M5/M6）

---

## 1. 目标与范围

### 1.1 总体目标
在保持现有内核能力（调度、恢复、可观测、MCP 集成）不倒退的前提下，优先提升：
1. 新用户上手速度
2. 日常使用体验一致性
3. 团队协作可见性与可运营性
4. 采用效果可量化

### 1.2 范围内
- CLI / REPL / Dashboard 的体验整合
- 上手路径、模板、配置引导、诊断提示
- 团队级协作视图与采用度量体系

### 1.3 非目标
- 新增大规模底层架构重写（如调度内核重构）
- 引入重型商业化计费系统
- 对外生态市场化（插件商店）作为本阶段核心目标

---

## 2. 设计原则

1. **体验一致性优先**：同一概念在 CLI / REPL / UI 中命名与行为一致。
2. **可解释优先**：错误信息必须给出可执行修复路径，而不只是报错文本。
3. **最小学习闭环**：10 分钟内完成“安装→配置→运行→观察→定位问题”。
4. **指标驱动迭代**：每月里程碑必须绑定采用与稳定性指标，不做无指标优化。
5. **不牺牲可靠性**：UX 增强不得破坏已有 guardrails、retry/fallback、审计链路。

## 3. 分阶段路线（M4-M6）

## 3.1 Phase M4（Month 1）：体验统一与可见性提升

### Task 4.1 统一交互入口（CLI + REPL 语义一致）
**目标**：降低命令心智负担，减少“同功能不同入口行为不一致”。

**设计要点**：
- 统一命令命名、参数优先级与帮助文案结构
- 明确 REPL 指令与 CLI 子命令映射关系
- 统一成功/失败输出结构（可机器解析 + 人类可读）

**验收（DoD）**：
- 命令语义对照表完整
- 核心路径回归通过（submit/approve/reject/exit/verbose）
- 无破坏性兼容问题（如有旧命令，提供明确迁移提示）

### Task 4.2 Dashboard 任务全景页可用化
**目标**：让用户一眼看懂任务生命周期与异常位置。

**设计要点**：
- 展示 TaskGraph 实时状态流转（节点状态、分支、join）
- 支持事件流筛选（task/node/type/time）
- 提供“关键节点跳转 + 上下文定位”能力

**验收（DoD）**：
- 单任务执行全链路可视化可复盘
- UI 展示与事件日志数据一致
- 异常节点可在 2 次交互内定位到原因上下文

### Task 4.3 可视化调试最小闭环
**目标**：把“运行后看日志”升级为“运行中可观察 + 可跳转”。

**设计要点**：
- CLI 输出 task_id 并可直达 Dashboard 详情
- 高亮关键异常：重试、fallback、guardrail 触发
- 保留终端最小摘要，避免 UI 与 CLI 信息割裂

**验收（DoD）**：
- 至少 3 类关键异常可视化高亮
- CLI→UI 跳转链路稳定
- 问题定位耗时相较基线有明确下降

## 3.2 Phase M5（Month 2）：上手路径与采用率工程

### Task 5.1 Quickstart 2.0（10 分钟跑通）
**目标**：新用户不依赖口头指导即可完成首个成功任务。

**设计要点**：
- 提供 simple / parallel / tool-call 三类最小示例
- 提供 preflight 检查（环境、配置、关键依赖）
- 明确“失败时下一步”而非仅输出错误

**验收（DoD）**：
- 新用户按文档 10 分钟可完成首跑
- preflight 覆盖 Top 常见环境问题
- 文档与实际命令零偏差

### Task 5.2 配置向导与错误可解释化
**目标**：降低配置门槛，减少重复支持沟通。

**设计要点**：
- 配置向导覆盖模型、fallback、并发、预算关键项
- 常见失败给出“原因 + 修复命令/路径 + 验证方式”
- 错误分级（阻断/警告/信息）统一呈现

**验收（DoD）**：
- Top N 错误具备可执行修复建议
- 初次配置成功率明显提升
- 用户无需查源码即可完成主路径修复

### Task 5.3 场景模板与最佳实践包
**目标**：把可复用经验产品化，降低“从零搭建”成本。

**设计要点**：
- 三类模板：研究型、执行型、多 Agent 协作型
- 模板内置 guardrails/evaluator 推荐配置
- 每个模板含示例输入、预期输出、常见陷阱

**验收（DoD）**：
- 模板可一键运行
- 模板文档覆盖“如何改造为自己的任务”
- 模板使用率进入月度追踪

## 3.3 Phase M6（Month 3）：团队协作与采用度量

### Task 6.1 团队工作视图
**目标**：从“个人运行”升级为“团队可追踪协作”。

**设计要点**：
- 按成员/任务组展示执行状态与结果分布
- 提供跨任务对比（成功率、失败类型、耗时）
- 支持团队周会复盘视图导出

**验收（DoD）**：
- 团队可识别高频失败模式与瓶颈环节
- 支持按时间窗查看趋势

### Task 6.2 协作治理最小闭环
**目标**：在不增加流程摩擦前提下建立必要治理。

**设计要点**：
- 高风险操作确认策略可配置
- 关键操作审计可追溯（who/when/which task/action）
- 与现有流程兼容，不强制复杂审批链

**验收（DoD）**：
- 协作场景关键操作可审计
- 高风险操作具备显式确认与留痕

### Task 6.3 Adoption 指标体系
**目标**：让后续路线由数据驱动而不是主观判断。

**设计要点**：
- 核心指标：周活用户、任务成功率、平均恢复时长、模板使用率
- 建立月度复盘看板与阈值告警
- 指标定义与采集口径固定，避免月间不可比

**验收（DoD）**：
- M6 末可产出首份月度采用报告
- 指标可直接支撑下一季度优先级排序

## 4. 架构与数据流（高层）

1. **入口层（CLI / REPL）**：统一命令与输出协议，产出一致的 task 上下文标识。
2. **运行层（Runtime / Orchestrator）**：复用现有核心能力，不引入破坏性变更。
3. **观测层（EventBus / EventLog / Dashboard）**：形成“事件→可视化→定位→反馈”的闭环。
4. **产品层（Quickstart / Templates / Guides）**：将可用能力包装为可采用能力。
5. **运营层（Metrics / Team View）**：将使用行为转化为决策数据。

---

## 5. 错误处理与风险控制

### 5.1 主要风险
- UX 增强导致行为不一致或破坏现有脚本兼容性
- Dashboard 与日志口径不一致造成误判
- 指标口径不统一，导致路线判断失真

### 5.2 控制策略
- 关键命令建立兼容映射与弃用提示窗口
- UI 展示字段与事件 schema 做对齐校验
- 指标字典先行（定义、采集点、计算方式、刷新频率）
